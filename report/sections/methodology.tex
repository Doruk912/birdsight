\section{Methodology}

This section describes the methodology of the project in detail. The stages of
development from start to finish are explained, including strategies, design
choices, development lifecycles, and technological specifics.

The main goal of this project was to promote citizen science by building a platform
with an integrated machine learning model for birdwatchers and enthusiasts. The
project was divided into two separate phases. These phases consisted of the machine
learning phase of the project and the web platform development phase. The project
followed a structured methodology that combined a comparative analysis of different
machine learning models and applied software development.

The machine learning phase included algorithm selection, data collection, model
development, and evaluation. On the other hand, the web platform development phase
included the design and development of the website, integration of the machine
learning model, and testing.

\subsection{Machine Learning Model Development and Comparative Analysis}

The first main phase of the project was the development of machine learning models
for bird species identification. This phase included data acquisition,
preprocessing, model selection, training, and evaluation. The details of each step
are explained in the following subsections.

\subsubsection{Data Acquisition}

When developing a machine learning model, one of the most important requirements is
the availability of a large, high-quality dataset to train the models properly. In
this project, multiple models were evaluated, and high performance was expected from
them. To achieve accurate results, large numbers of bird images taken from different
angles were required. Furthermore, the data had to be labeled with the correct bird
species for the models to be trained effectively.

Because tens of thousands of bird images were required, it was not feasible to
create such a dataset manually. Instead, existing data was gathered. GBIF, which
stands for Global Biodiversity Information Facility was used to construct the
dataset. It is an open-access biodiversity platform that collects species
observation records from various sources \cite{gbif2025}. These sources are mostly
citizen science platforms such as iNaturalist \cite{inaturalist2025} and
Observation.org \cite{observation2025}. This once again demonstrates the importance
of citizen science. A single organization would not be able to maintain records of
the increasing number of species from every part of the world. Alternatively,
citizens from around the world keep records of species around them by uploading
observations. As people use these platforms, massive datasets are created that would
be almost impossible for a single organization to construct independently.

In this project, GBIF was used to construct a dataset of bird images. In order to
limit the scope of the project, bird species that are commonly observed in Turkey
were selected. The species were selected with reference to observation counts in
Turkey \cite{ebird2025}. Initially, the constructed dataset included 23 bird
species commonly observed in Turkey. The dataset consisted of 5000 images for each
species and a total of 115000 images.

The metadata for the selected species was downloaded from GBIF. Only records that
included an image and reliable species labels were used. The data downloaded from
GBIF was compiled from many different sources and included metadata such as species
name, location of observation, datetime of observation, observer information,
observation license, life stage of the organism, and most importantly, multimedia
related to the observation.

For each selected bird species, the last ten thousand observations were downloaded
from GBIF. Then, using a Python script \cite{python2025}, the images were downloaded
into their respective folders in Google Drive \cite{googledrive2025}. A dataset
containing bird images labeled according to their species folders was constructed.

\subsubsection{Data Preprocessing}

The collected image dataset was preprocessed to ensure that it was suitable for
training the machine learning models. Initially, low-quality and corrupted images
were removed from the dataset and replaced to keep class balance. The images were 
then resized to a consistent size to match the input dimensions required by the 
selected deep learning architectures.

To increase dataset variability and improve model generalization, several data
augmentation techniques were applied \cite{shorten2019survey}.
In image classification, augmentation generates additional training views by applying
label-preserving geometric and photometric transforms to existing images.
This expands the diversity of examples the optimizer sees without changing class labels or
collecting new photographs, which helps reduce overfitting when the number of raw images per 
class is finite. The offline pipeline used in this project included random rotations, 
horizontal flipping, brightness adjustments, and other minor transformations so that models 
become less sensitive to lighting, viewpoint, and framing differences typical of amateur bird 
photographs.

During training, additional stochastic augmentations (random resized crop to the 
target size, flips, colour jitter, etc.) were applied on-line to further improve robustness 
to viewpoint and lighting. Validation and test splits used only the deterministic preprocessing 
described above so that metrics remain comparable across models.

After preprocessing was completed, the dataset was divided into training,
validation, and testing subsets using a fixed splitting strategy. A total of
80\% of the dataset was reserved for training, while the remaining 20\% was
equally divided between validation and testing subsets.

\subsubsection{Model Selection}

During the literature review, studies related to bird species identification were
analyzed. In accordance with the findings obtained from the literature, model
selection in this project focused on deep learning architectures. Traditional
machine learning algorithms such as Support Vector Machines (SVMs) and Random
Forests generally underperform compared to deep learning models and are unable to
achieve the desired accuracy levels for fine-grained image classification tasks
\cite{lolge2025evolution}. Since this project focused on image-based bird species
classification, models that perform effectively on visual data were required.

This project focused on deep learning-based Convolutional Neural Network (CNN)
architectures that are widely used for image classification tasks. CNN-based
models are more suitable for identifying species from images because they are
capable of extracting complex hierarchical visual features such as texture,
shape, colour, and local patterns \cite{chen2024fetfgvc}. The following architectures
were selected because they frequently appear in the literature and have
demonstrated strong performance in image classification tasks.

\begin{itemize}

\item \textbf{VGG16:} VGG16 is a deep convolutional neural network with a relatively
simple and uniform architecture composed of stacked convolutional layers followed
by fully connected layers. Although it contains a large number of parameters, it
has been widely used in image classification tasks due to its strong feature
extraction capability \cite{bilgin2022bird,tang2025comparative}.

\item \textbf{ResNet50:} Residual Networks (ResNet) utilize residual connections that
allow deeper neural networks to be trained with improved gradient flow. In this
project, \textbf{ResNet50} was used as the representative ResNet depth because it is
widely reported in bird-species benchmarks \cite{bilgin2022bird,tang2025comparative}.

\item \textbf{InceptionV3:} This architecture utilizes parallel convolutional
filters of different sizes within the same layer, allowing it to capture
multi-scale visual features. This design is effective for bird species
classification because both overall shape and fine local patterns are important
for distinguishing visually similar species \cite{tang2025comparative}.

\item \textbf{MobileNetV2:} MobileNetV2 was designed for computational efficiency
through the use of depthwise separable convolutions. Although lightweight, it
has demonstrated competitive performance in similar image classification tasks
\cite{lolge2025evolution,tang2025comparative}.

\item \textbf{DenseNet121:} DenseNet121 connects each layer to every other layer in
a feed-forward manner. This structure promotes feature reuse and improves
gradient flow throughout the network. Comparative studies have shown that
DenseNet architectures often outperform traditional CNN models
\cite{bilgin2022bird}.

\end{itemize}

Each selected model was trained and evaluated using the same dataset and
experimental conditions to ensure a fair comparison between architectures. The
performance of each model was evaluated using multiple metrics, including
accuracy, precision, recall, F1-score, and inference speed.

\subsubsection{Model Training}

All models were trained using the Python programming language
\cite{python2025}. Python was selected because of its large ecosystem and
extensive support for machine learning and deep learning applications.
Specifically, the PyTorch deep learning framework was used
for model development and training \cite{paszke2019pytorch}.
PyTorch was selected because it provides built-in support for
transfer learning and implementations of widely used convolutional neural
network architectures. The implementations of VGG16, ResNet50, InceptionV3,
MobileNetV2, and DenseNet121 provided by Torchvision \cite{torchvision2025} were used to ensure
consistency between experiments. The overall workflow used during model training and 
evaluation is illustrated in
Figure~\ref{fig:training_pipeline}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\linewidth]{images/training_pipeline.png}
    \caption{Overview of the model training and evaluation pipeline}
    \label{fig:training_pipeline}
\end{figure}

Each selected architecture was initialized with pretrained ImageNet weights.
This is a common practice in similar bird species classification studies
\cite{bilgin2022bird,tang2025comparative,lolge2025evolution}. Transfer
learning significantly reduces training time and improves model convergence,
especially when working with limited computational resources. The original
classification layers of each model were replaced with new fully connected
layers corresponding to the number of bird species included in the dataset.

Training was performed in two phases. In the first phase, the pretrained backbone 
layers were frozen and only the newly added classification layers were trained. 
In the second phase, all layers were unfrozen and the entire network was fine-tuned 
using differential learning rates. Phase~1, which is the frozen backbone, optimised only 
the new classification head with a learning rate of $10^{-3}$ and weight reduction of
$10^{-4}$, except for VGG16, where weight reduction of $5\times 10^{-4}$ in code 
to stabilise the larger fully-connected stack. Phase~2, which is the unfrozen backbone, used two 
sets of parameters: $10^{-5}$ for convolutional backbone layers and $10^{-4}$ for 
the classifier header. The backbone should move slower than the randomly initialized 
header because it already encodes general visual features from ImageNet \cite{imagenet2025}.

Images were preprocessed using resizing and normalization techniques. 
Random resized cropping, horizontal flipping, and color jittering were also applied to
the training images in order to improve the generalization performance of models and reduce overfitting.

Supervised learning was used during model training. Input images and their
corresponding species labels were provided to the models in batches. During
forward propagation, the models generated predicted class probabilities for
each input image. The cross-entropy loss function was used because the task 
involved multi-class image classification. Model parameters were updated using 
the backpropagation algorithm together with the Adam optimization algorithm.

Training followed a fixed schedule in the code for each model. In Phase~1, the 
convolutional backbone stayed frozen while only the new bird-species classifier was 
trained. MobileNetV2, ResNet50, DenseNet121, and InceptionV3 used 5 frozen epochs; 
VGG16 used 10 frozen epochs so its larger classifier had more time to settle before 
the backbone was unfrozen. In Phase~2, all weights were updated, with at most 20 
fine-tuning epochs as a hard upper limit. Inside each phase the learning rate followed 
a cosine decay curve, and gradient norms were clipped at 1.0 for stability. Early 
stopping applied only in Phase~2: if validation loss did not improve by at least 
0.001 for 7 epochs in a row, training stopped even if 20 epochs had not been reached.

After every epoch the script measured loss and accuracy on the validation split that 
was not used for training. The training loss is the usual multi-class cross-entropy 
between the model's softmax probabilities and the true species label; InceptionV3 
experiments that add label smoothing are described later in this chapter. Validation 
accuracy is simply the fraction of validation images whose predicted species matches 
the label.

Model checkpoints were automatically saved whenever validation accuracy improved. This
ensured that the best-performing versions of the models were preserved for evaluation.

To ensure a fair comparison between architectures, the same dataset splits
were used for all experiments. Batch sizes were selected according to the memory requirements 
of each architecture.
The models were then evaluated using consistent evaluation metrics, and all
experimental settings and configurations were documented to ensure reproducibility.
The main training hyperparameters used for each model are summarized in
Table~\ref{tab:training_hyperparameters}.

Training of the models was performed on a single NVIDIA RTX 4050 Laptop GPU with CUDA 
acceleration enabled \cite{nvidiacuda2025}.
Images were loaded and batched using the PyTorch DataLoader with shuffling on 
the training split \cite{paszke2019pytorch}. On-the-fly transforms then produced the 
architecture-specific tensors described above: \textbf{$299\times 299$} for InceptionV3 
and \textbf{$224\times 224$} for the other 4 models.
 
InceptionV3 was designed around a $299\times 299$ field of view, whereas the VGG, ResNet, 
DenseNet, and MobileNet implementations used here follow the common $224\times 224$ 
ImageNet recipe. Keeping the same resize-and-crop convention during training, evaluation, 
and (for InceptionV3) deployment inference avoids a train--serve mismatch and aligns filter 
statistics with the pretrained weights \cite{paszke2019pytorch}.

\subsubsection{Model Evaluation}

After completing the training phase, all developed models were evaluated in
order to compare their performance. The aim for this phase was not only to 
determine the accuracy, but also to analyze the practical suitability of each 
architecture for deployment in a real world application.

Each model was evaluated using the test split that was created during the preprocessing stage. 
The models were never trained on the test data. Multiple evaluation metrics were examined in order
to provide a comprehensive comparison between architectures.

\paragraph{Evaluation Metrics}

Several evaluation metrics were used to measure model performance. These metrics included accuracy, precision,
recall, and F1-score.

Accuracy measures the percentage of correctly classified images among all
predictions. It provides an overall indication of model performance.
However, accuracy alone does not fully represent the model's quality,
especially in a fine-grained image classification task where most classes are visually similar.

Precision measures the proportion of correct positive predictions among all
positive predictions generated by the model. High precision indicates that
false positive predictions are relatively low.

Recall measures the proportion of correctly identified samples among all
instances of a class. High recall indicates that the model is capable
of identifying most true samples and minimizes false negatives.

F1-score is the harmonic mean of precision and recall. It provides a balanced
evaluation metric that considers both false positives and false negatives.

In addition to the standard classification metrics, Top-3 Accuracy and
Top-5 Accuracy were also evaluated. These metrics measure whether the correct
bird species appears within the top three or top five predictions generated
by the model. 

Receiver Operating Characteristic Area Under Curve (ROC-AUC) and mean Average
Precision (mAP) scores were also computed to evaluate the discriminative
capabilities of the models across all classes.

Inference speed was also considered during evaluation because practical
deployment requires not only accurate but also efficient models.

\paragraph{Evaluation Process}

The evaluation process was conducted by recording the predictions of each trained 
model on the test dataset which models were not trained on. 
The models' predicted labels were compared with the actual labels to generate 
confusion matrices and calculate all evaluation metrics.

For the sake of consistency and reproducibility, all evaluations were performed using the
same hardware and software environment. The performance of each architecture was recorded.

\subsubsection{Final Model Optimization}

In the previous section, the InceptionV3 model was selected as the best-performing model.
In order to further improve the model's performance, additional experiments were conducted.
Although the model already achieved good results, we believed that the model's generalization
ability and overall accuracy could be further improved.

Three separate experiments were conducted using the same 23-class dataset to optimize the model. 
Each experiment independently focused on a different aspect. The best-performing configurations 
were identified and combined to achieve the best possible results. The final training script was 
then used 
to train the model on the expanded 30-class dataset.

\paragraph{Label Smoothing}

The first experiment was applying label smoothing during training. Label smoothing is a regularization
technique used to replace the hard class labels with smooth values. This helps to prevent overfitting 
and improve generalization performance.

In this experiment, a label smoothing factor of 0.1 was applied to the cross-entropy loss function. 
This means that instead of the correct class having a label of 1 and all other classes have a label 
of 0, 
the correct class was assigned a label of 0.9 and the remaining classes were assigned a label of 0.1 
divided by the number of classes minus one.

The experiment resulted in a small improvement in accuracy. The overall accuracy increased from 89.10\% 
to 89.49\%.
The weighted F1-score also improved slightly. However, the improvement was limited compared to the 
baseline.

\paragraph{Enhanced Data Augmentation}

The second experiment was about improving the data augmentation techniques used during training.
The initial data augmentation already included the standard augmentations such as random resized 
cropping,
horizontal flipping, and color jittering. However, it was believed that applying more aggressive 
augmentations could further improve the model's performance.

The baseline augmentation pipeline consisted of:

\begin{itemize}
    \item Random resized cropping
    \item Random horizontal flipping
    \item Color jittering
\end{itemize}

The enhanced augmentation pipeline additionally introduced:

\begin{itemize}
    \item Wider random resized crop scaling ranges
    \item Random vertical flipping
    \item Random image rotation
    \item Random perspective distortion
    \item Stronger brightness, contrast, saturation, and hue variation
    \item Random grayscale conversion
    \item Random erasing for simulated occlusion
\end{itemize}

Applying more aggressive augmentation better simulated the amateur bird photographs that the model 
would encounter in real-world scenarios. This experiment resulted in the largest performance gain 
among all experiments. Test accuracy increased from 89.10\% to 89.64\%, while Top-5 accuracy improved 
from 97.82\% to 98.03\%.

Additionally, the training curves showed improved generalization performance. In the initial training 
curve, the training accuracy exceeded the validation accuracy. This indicates that the model was 
overfitting to the training data. With the enhanced augmentation, the validation and training curves 
were closer together and the training curve never exceeded the validation curve. This shows that the 
heavier augmentation helped reduce overfitting and improved the model's ability to generalize unseen 
data.

\subsubsection{Final Model Training}

After completing the experiments for model optimization, the best-performing configurations were determined.
The final training script was written by combining the configurations. The final configurations include:

\begin{itemize}
    \item Label smoothing with a factor of 0.1
    \item Enhanced data augmentation
    \item 11 frozen backbone epochs before full fine-tuning
\end{itemize}

The combined script was first used to train the InceptionV3 model on the original 23-class dataset. 
The model achieved a test accuracy of 89.5\% and a validation accuracy of 90.28\% on the 23-class dataset, which was an improvement over the baseline.

The optimized training script was then used with the expanded
30-species dataset containing 150000 bird images. Expanding the dataset
increased the difficulty because the model had to distinguish between 
a larger number of visually similar species.

Despite the increased difficulty, the final model maintained strong
overall performance. The final InceptionV3 model achieved an overall
accuracy of 89.35\%, Top-3 accuracy of 95.98\%, and Top-5 accuracy of
97.80\%. The model also achieved a weighted F1-score of 0.893 and a
macro ROC-AUC score of 0.996.

The final trained model was selected for deployment and integration into
the developed citizen science web platform because it provided the best
overall balance between classification accuracy and practical inference
performance.

    \caption{User interaction and observation submission workflow in BirdSight}
    \label{fig:user_workflow}

The web platform was developed following a three-tier architecture, 
consisting of a presentation layer, an application layer, and a data layer. 
This separation improves the maintainability of the project. It also aligns 
with best practices in modern web development \cite{cervantes2024software}. 

The platform consists of three independently deployable services: a Spring Boot backend \cite{springboot2025}, 
a Next.js frontend \cite{nextjs2025}, and a FastAPI machine learning service \cite{fastapi2025}. Helper services such as  
the database and object storage were containerized using Docker \cite{docker2025}.

\subsubsection{Backend Development}

The backend was developed using Java 21 \cite{oraclejava2025} with the Spring Boot 4 framework. Spring Boot 
was selected for its widespread usage in the industry and its convention-over-configuration
approach. Using Spring Boot simplified dependency management, service configuration, and 
deployment \cite{springboot2025}.

The backend exposes a RESTful API using Spring Web MVC \cite{walls2022spring}. This API serves as the only 
communication between the frontend and all server-side functionality. The functions
handled by backend include observation management, user authentication, taxonomy browsing, 
and machine learning model requests. JSON was used as the data exchange format due to 
its compatibility and lightweight nature \cite{bray2017json}.

The application logic is organized into distinct packages. Separate packages handle 
authentication, user management, observations, community identifications, comments, 
taxonomy, machine learning, and object storage. This modular structure keeps 
responsibilities separated and makes the codebase easier to maintain.

Authentication was implemented using a dual-token strategy \cite{rfc7519jwt}. When the user logs in, the backend 
issues a short-lived JWT access token returned in the response body and a long-lived 
refresh token stored in a cookie. The access token has a lifetime of only 15 minutes.
The refresh token on the other hand remains valid for 7 days.

This design ensures that the access token is never persisted in the browser storage.
This reduces the risk of cross-site scripting attacks. When an access token expires, 
the frontend automatically uses the refresh token cookie to get a new access 
token from the backend. This happens without requiring the user to log in again. 
Spring Security was used to configure the security filter chain and enforce 
authorization rules across all endpoints \cite{manne2023springsecurity}.

Database access was abstracted using Spring Data JPA with Hibernate \cite{tudose2021orm}. This reduced 
boilerplate code. Database schema management was handled by Flyway \cite{flyway2025}. Flyway ran migrations 
automatically on application startup. The migrations include all necessary table creation and schema updates.
Flyway ensured that the database schema is always up to date with the application code. It also provided a version-controlled 
way to manage database changes across different environments.

A quality grading system for observations was implemented in the backend. Each observation 
is assigned one of two quality grades: \textit{Needs ID} or \textit{Research Grade}. 
The grade is recalculated every time an identification is submitted to an observation.
The system determines a community taxon which is defined as the lowest taxon in the taxonomy 
tree agreed by more than two-thirds of identifications. If the community taxon is at 
the species level and at least two users have agreed, the observation is marked as Research Grade.

This grading system is implemented in order to add a quality control mechanism on top of the
machine learning model. It encourages users to participate in the identification 
process and helps to ensure that the data collected through the platform is of high quality.

Reverse geocoding was integrated into the backend using the OpenStreetMap Nominatim API. This is
implemented to automatically convert the GPS coordinates for each observation into human-readable
location names \cite{nominatim2025}.

\subsubsection{Database Design}

PostgreSQL was chosen as the database for this platform because of its adoption in the industry and its
open-source nature \cite{postgresql2025}. On top of PostgreSQL, PostGIS was used for geospatial data support. 
This allowed storing and querying observations based on their geographic location \cite{postgis2025}.

The database schema includes tables for users, observations, taxonomy, identifications, comments, and images.
Foreign keys preserve the integrity of data. Indexes are also used for improved performance.

The taxonomy table is unique as it stores the hierarchical structure of bird taxonomy. 
Each taxon has a parent-child relationship with other taxa. This allows efficient querying of related species.
The table stores all bird taxa from family level all the way down to species level. The taxonomy data was sourced
from IOC World Bird List \cite{ioc2025}. The table contains about 14000 rows representing all bird taxa.
The schema for the taxonomy table is shown in Table~\ref{tab:taxa_schema}.

Database changes are applied with Flyway. Flyway runs SQL migration files when the backend starts. 
Each file is like a version step, so the schema is repeatable on another computer or later in time \cite{flyway2025}.

All images that the users upload are stored in MinIO object storage \cite{minio2025}. This includes the images that the users upload as observations or their avatars. 
The database only stores links to the images. MinIO is an S3-compatible open-source storage solution. 
The backend handles image uploads, deletions, and retrievals.

Next.js was selected for its built-in support for server-side rendering and API routes. Next.js also provides a great developer experience with features like hot reloading.
TypeScript was used to add static typing which improves code quality and maintainability \cite{typescript2025}.

The service layer in the frontend is organized into separate modules for each domain. This includes authentication, observations, taxonomy, machine learning, and user management.

All API calls are routed through a Next.js rewrite proxy, so the frontend never exposes the backend host or port directly to the browser.

\subsubsection{Map Integration}

Visualizing the observation on a map is an important feature of the platform. It allows users to explore the distribution of sightings. The map was implemented using MapLibre GL 5 together with the react-map-gl library \cite{maplibre2025, reactmapgl2025}.
For the visuals on the map, the platform uses Carto's Voyager tile style \cite{carto2025}. This relies on crowd-sourced geographical data from OpenStreetMap \cite{openstreetmap2025}.

The map dynamically adjusts rendering based on the zoom level. When zoomed in, the map shows individual markers for observations however when zoomed out, the markers are clustered. This clustering is implemented to improve performance and user experience.
Interacting with these markers triggers pop-ups displaying observation cards. These cards include details like the species name and a thumbnail image.
There is also a sidebar that displays a filterable list of all observations. Additionally, the user can draw a bounding box on the map to filter observations within that area.

The backend provides a dedicated API endpoint that returns only the coordinates and basic information to improve performance.

\subsubsection{Machine Learning Model Integration}

The trained InceptionV3 model was deployed as a separate microservice using Python \cite{python2025} and FastAPI \cite{fastapi2025}. This service exposes an API endpoint that expects an image and returns the predicted bird species along with confidence scores.
Separating the machine learning model into its own service allows for better scalability and maintainability. The model can be updated or retrained independently without affecting the rest of the application.
This is a recommended practice for integrating machine learning models into web applications \cite{karanam2026microservice}.

The service exposes a single POST endpoint at /predict. This endpoint accepts an image file and then performs the same preprocessing steps as during training. 
The image is resized to 299x299 pixels, normalized, and then passed through the InceptionV3 model to get predictions.
The service returns a JSON response \cite{bray2017json} containing the top 5 predicted species along with their confidence scores.

Only the Spring Boot backend \cite{springboot2025} communicates with the machine learning service. The frontend never interacts with the machine learning service directly. 
When a user uploads an observation or suggests an ID, the backend sends the image to the machine learning service and retrieves the predictions.
This is done to abstract away the machine learning functionality and to ensure that the frontend only interacts with a single backend API.

When the user is uploading an observation or adding an identification, the machine learning suggestion pop-up appears.
This pop-up shows the predictions from the model. How many prediction results are shown depends on the confidence scores. 
If the model is highly confident only the top prediction is shown. If the confidence scores are lower the top 3 or all 5 predictions are shown.

By default, the confidence score percentages are hidden to avoid overwhelming users with technical details. However there is a toggle  that allows users to view details.
